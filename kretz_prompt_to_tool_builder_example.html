Two prompts. Fifteen minutes. One working tool.

Timber Sale Tracker — built from a description.

The follow-along example for the final activity. It solves a plain, everyday problem — timber sale follow-ups that get lost in notebooks and inboxes — and it was built with two prompts and nothing else. No coding, no software to buy, no login. The point is not this particular tracker. It is that describing a tool well is a skill anyone in the room can use on their own annoying task.

Why this problem. Land, sales, and forestry work is full of follow-ups that live in one person's head or a paper notebook — who was called, who got walked, whose bid is still out. That is a small, clear, low-risk problem, which makes it a good first tool. Everything below runs from a double-clicked file, needs no internet, and required no purchased software or hosting to build.

The two prompts that built it.

This is the real payload of the activity. The tool is just proof the prompts worked.

Prompt 1 — the build
Build me a single-file HTML tool to help with this problem: our land, sales and forestry people lose track of timber sale follow-ups. Everything lives in someone's head or a notebook. It should let me add a prospect with landowner, town, acres, primary species, stage, next action, due date, forester, and notes. Stages run from contacted, to walked, to bid submitted, to under contract, to harvest scheduled, to closed. Show them in a table sorted by due date, with anything overdue highlighted. Let me filter by stage and by forester, and show me counts of open prospects, total acres, overdue follow-ups, and what is due in the next week. It needs a review warning making clear the tool does not verify ownership or acreage and does not estimate timber value. Add a CSV export and a print button. No external dependencies, no login, no internet connection. One file that runs from a double-click on a work laptop. Ask up to five questions before coding if you need to.
Prompt 2 — the refine
Two changes only. Load it with four sample rows so it is not empty when someone opens it, and label them clearly as sample data. And make the stage badges color-code so under contract and harvest scheduled stand out from the early stages. Leave everything else exactly as it is.
That is the whole build. Two prompts, in the room, start to finish. The takeaway is not this tracker — it is that a tool worth using can come out of a clear description, and the skill of writing that description is the same one the whole session has been practicing.

What it deliberately does not do.

Knowing where a tool stops is part of building one.

It does not save between sessions. Close the tab and the rows are gone — export the CSV. That is a real limitation of a fifteen-minute build, and naming it is more useful than hiding it. Saving your data between visits would be the natural next thing to add, and it makes a good follow-on exercise.

It does not talk to anything. No parcel data, no accounting, no email. If it needed to, that is the moment to ask whether this should be a real system instead of a one-file tool.

It is one small slice. It tracks the follow-up that gets dropped, not an entire sales process. Building the piece that annoys you most, first, is a good way to find out whether a tool is worth expanding before you put more time into it.
